#### 9.8.1.2 Thermal Event Detection (SOC_THERM_thermctl)
- SOC_THERM_thermctl monitors temperatures coming from the SOC_THERM_tsensor and takes
configurable actions at configurable temperature thresholds. The available actions are:
- Generating an interrupt
- Asserting one or more hardware thermal throttling indicators to SOC_THERM_throttlectl
- Alerting HSM
- Invoking thermtrip to shut down the system
There are essentially four major categories:
- Group-temperature-based thermal actions
1.
- Generic software trip
Heavy hardware throttling, asserted by rising temperature.
- HSM Alerts
2. HSM hot alert, asserted by rising temperature. HSM cold alert, asserted by falling temperature. Tj_max-based shutdown triggered by rising temperature, sent to PMC, HSM, and PSC. Tj_min-based shutdown triggered by falling temperature sent to PMC, HSM, and PSC. 3. 4. Actions are only taken for thresholds which are enabled and have a valid temperature input.
- Group-Temperature-Based – Category 1
The group-temperature-based thermal actions (category 1) are organized as a two-dimensional array – thermal groups (horizontally) and response types (vertically). Each intersection of a group and response type has a monitor. Each monitor includes its own thermal threshold, and all the monitors act independently of each other. The following options are available for each monitor in category 1: EN – should the monitor operate or not? A monitor will operate if this bit is enabled AND the temperature input is valid.
- UP_THRESH – A temperature threshold. For hardware throttling monitors, temperature rising
beyond this threshold causes the monitor to assert its hardware throttling output.
- DN_THRESH – A temperature threshold. For hardware throttling monitors, temperature falling
below this threshold causes the monitor to deassert its hardware throttling output. If the down interrupt is enabled, the down interrupt is asserted.
- INTR_UP – should the monitor raise an interrupt when temperature rises above UP?
- INTR_DN – should the monitor raise an interrupt when temperature falls below DN?
A monitor includes two separate Finite State Machines (FSMs) for generating the up and down interrupts. The state machines implement hysteresis. So, for example, once a down interrupt has been asserted and cleared it won’t be reasserted until after the temperature has risen above UP.

- Thermal Sensor and Thermal Throttling Controller
Similarly, once an up interrupt has been asserted and cleared it won’t be reasserted until after temperature has fallen below DN. Software can force a transition back to the idle by writing to the monitor’s control register.
- Conceptually, a monitor asserts its output indicator based on the state of the FSM which
generates the up interrupt. Consequently, the monitor cannot request cold throttling (i.e., throttle below a particular temperature). If cold throttling is required, it must be implemented in software.
- HSM Alerts – Category 2
The HSM alerts, unlike the group-temperature-based thermal actions, only use Tj_max and Tj_min respectively. They are not replicated per group, since HSM does not need to know which group is getting too hot or too cold.
- Other than not being replicated per group, the HSM hot alert works the same as the ones in
category 1 and use the same kind of FSM. The options available for HSM cold alert monitors are very similar, but the output is asserted on falling temperature instead of rising temperature. The behavior of the UP_THRESH and DN_THRESH changes for the HSM cold alert monitor instances:
- UP_THRESH – A temperature threshold. Temperature rising beyond this threshold causes the
monitor to deassert the HSM cold alert output. If the up interrupt is enabled, the up interrupt is asserted.
- DN_THRESH – A temperature threshold. Temperature falling below this threshold causes the
monitor to assert the HSM cold alert output. If the down interrupt is enabled, the down interrupt is asserted.
- Tj_max / Tj_min Shutdown – Categories 3 and 4
- The thermal response logic for categories 3 and 4, which respectively correspond to high
temperature and low temperature shutdown, is much simpler. Each has an enable toggle and a single temperature value. If the temperature exceeds the overtemperature limit while the trip is enabled and has a valid temperature reading, the chip is shut down. The same goes for the under- temperature limit, except it triggers then the temperature goes below that programmed limit. SOC_THERM implements the thresholding but relies on PMC to implement the shutdown. Thermtrip is the ultimate backstop for excessively high or excessively low temperatures. It should only happen in truly exceptional circumstances (when normal thermal management fails to control temperatures). A thermtrip shutdown is NOT graceful; it may cause storage media corruption and it provides no mechanism for alerting the user prior to the shutdown. The probability of corrupting storage on a thermtrip may be low, but it is non-zero. The only way to mitigate the issue is to avoid a thermtrip shutdown in the first place.
##### 9.8.1.2.1 Interrupt Handling
- The SOC_THERM thermal interrupt is the aggregation of N interrupt event sources. N is the
product of X groups, two levels, and two directions (up and down) plus interrupts for HSM hot alert and HSM cold alert. Event aggregation and software interrupt management is managed via an

interrupt status register INTR_STATUS and an interrupt enable register pair INTR_EN / INTR_DIS. The status field is set, regardless of the state of the enables, when temperature breaches the corresponding threshold. Software can read to understand which group and which type to service.
#### 9.8.1.3 Electrical Event Detection (SOC_THERM_edp)
- SOC_THERM_edp centralizes a few mechanisms for detecting and responding to externally
signaled electrical events, usually over-current events. SOC_THERM_edp can be configured to raise interrupts when these events occur. It can also assert a throttle indicator to SOC_THERM_throttlectl for each event. The assertion of a throttle indicator may lead to hardware throttling of CPU or GPU clocks.
##### 9.8.1.3.1 External Over-Current Detection
Platform power delivery hardware has knowledge of over-current or voltage-droop events which cannot be easily detected by the SoC. SOC_THERM accepts multiple signals from platform level hardware indicating that an event is occurring. These are known as OC alarms because they often indicate an over-current event. Software can configure the SOC_THERM response to a particular OC alarm being asserted. In particular, software can configure SOC_THERM to initiate throttling in response to an OC alarm assertion. SOC_THERM hardware can detect the signal and throttle much faster than software could detect and respond to a GPIO interrupt. SOC_THERM provides the following set of capabilities for each over-current alarm:
- Sticky and Brief Throttling Modes: Software can configure the OC alarm response to be
either sticky or brief. 1. Sticky Throttle Mode:
- Applicability: This is used for OC events that need a longer-term frequency adjustment
(for e.g., ac-adapter unplug event).
- Behavior: On the assertion of over-current signal, hardware asserts the corresponding
- OC_ALERT signal to engage throttling. Hardware sends an interrupt to software and
remains in the throttled state until software responds.
- Software Setup: The amount of throttling applied on an over-current event is
preprogrammed by software as appropriate for the type of over-current event that are being detected. The throttling point may be fixed or may need to be updated along with standard DFS clock changes.
- Software Response to Over-current Event: Events requiring sticky throttling are typically
those that require a longer-term adjustment to clock frequencies. After an over-current event, software reprograms clocks to appropriate levels, disengages throttling, and re- arms the over-current feature with new settings. Note that re-arming may first depend on detection of an over-current deassertion event (e.g., if over-current signal is really an
- AC-mode signal). These events may also require software to switch to more conservative
DVFS and EDP tables to prevent repeated occurrences of this event. - - - - Brief Throttle Mode:

- Thermal Sensor and Thermal Throttling Controller
-
- Applicability: This is used for transient OC events that need brief frequency adjustments
(e.g., voltage droop detected by a voltage comparator).
- Behavior: On the assertion of over-current signal, hardware asserts the corresponding
- OC_ALERT signal to engage throttling. Throttling remains engaged for a preprogrammed
amount of time after the over-current signal deasserts. Optionally, software can be interrupted on the event. This is implemented using a simple timer which controls the assertion of the related OC_ALERT signal that goes to SOC_THERM_throttlectl block.
- Software Setup: Software needs to set up two things: how much to throttle and how long
to throttle beyond the OC-event deassertion. Then once armed, software need not do anything else. The purpose is to slow the system down for a period of time to cool things off before returning to normal operation.
- Software Response to Over-current Event: Once armed, software doesn’t need to do
anything. Software can be optionally interrupted after a certain number of these events have occurred. Repeated occurrences may indicate that software needs to switch to a more conservative frequency capping policy. - - - 2.
- Configuration Registers: Software is responsible for configuring the behavior of each OC
alarm. SOC_THERM itself does not have any hardware mechanism to distinguish between the different OC alarms.
- Interrupt Control/Enable/Disable/Status Registers: Each OC alarm is associated with an
interrupt threshold register and software is interrupted when this threshold is breached (if its set to 0, interrupt is disabled, set to 1 means software is interrupted on every event). 3.
##### 9.8.1.3.2 Throttling Hardware Requirements
The over-current alarm signal is routed to the Thermal Throttling Controller (SOC_THERM).
- Depending on the OC alarm configuration, SOC_THERM_edp controls the corresponding OC_ALERT
indications to the SOC_THERM_throttlectl block.
- In brief throttling mode, a programmable timer (THROTTLE_PERIOD) is needed to control the
corresponding OC_ALERT signal. The timer is enabled when the external OC alarm is deasserted. The OC_ALERT to SOC_THERM_throttlectl is asserted until the timer expires. In sticky throttling mode, the OC_ALERT to SOC_THERM_throttlectl block remains asserted until software can respond.
#### 9.8.1.4 Throttling Controller (SOC_THERM_throttlectl)
Clock slowdown is the primary response method for Thermal and EDP management.
- SOC_THERM_throttlectl performs the mapping of thermal and electrical events to different
throttling clock throttling levels. CPU and GPU are the two clocks which receive such throttle indicators. The outputs of the throttlectl block are separate 3-bit throttle indicators to the CPU and GPU. These three bits represent different throttling levels in a thermometer style to prevent glitches. 3’b000: No throttling (Default)

- Thermal Registers
3’b001: Light throttling 3’b011: Medium throttling 3’b111: Heavy throttling It is the decision of the respective clock domains in terms of what to do with the signals. For example, the respective clock domains will need to make sure to update their clocks in a safe manner. Ultimately, they decide how to go from light/medium/heavy to an actual frequency value.

FSI_THERM. FSI_THERM aids in the thermal management of Functional Safety Island (FSI). It processes data from the thermal sensors placed within FSI and signals events based on software configuration. These events are used by software to take appropriate actions (i.e., reset by PMC, Error Handling by Safety CPU) based on the nature of the event generated. FSI houses two thermal sense macros. #### 9.8.1.5 FSI_THERM
- There is another instance of SOC_THERM located in the Functional Safety Island named
FSI_THERM. FSI_THERM aids in the thermal management of Functional Safety Island (FSI). It processes data from the thermal sensors placed within FSI and signals events based on software configuration. These events are used by software to take appropriate actions (i.e., reset by PMC, Error Handling by Safety CPU) based on the nature of the event generated. FSI houses two thermal sense macros. FSI_THERM compares the two sensors for consistency and notifies an error if the difference between the sensors is beyond the programmed temperature range.}.Body)
