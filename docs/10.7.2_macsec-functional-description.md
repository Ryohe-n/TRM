# 10.7.2 MACsec Functional Description

@{Key=10.7.2; Title=MACsec Functional Description; Body=The MACsec IP consists of three major blocks: MACsec Tx, MACsec Rx, and the Configuration block. MACsec Tx provides Global Bypass, if needed, to bypass MACsec Tx logic (Egress direction). Similarly, MACsec Rx provides the Global Bypass, if needed, to Bypass MACsec RX logic (Ingress direction). MACsec also support loop-back where Egress traffic from MACsec can be looped back to Ingress MACsec.

- Packet type: Normal Packet or preempted packet

- MACsec ICV GENERATION
The next step after Egress SC lookup stage is ICV generation. AES_CORE for ICV generation requires the following inputs. SAK[255:0] derived from key LUT. The TZ app programs this field. H[127:0] derived from LUT. H=AES(SAK,0) The TZ app programs this field.
- Additional Authenticated Data (AAD) which is packet data from MAC-DA, MAC-SA, SecTAG,
and user data.
- IV = {SCI, PN}. The 64bit SCI is derived from Egress SC Parameter Table and 32bit PN is
derived from Egress SA state Table.

- MACsec Functional Description
- IV is different between two calls to the AESGCM Engine, as every ethernet frame uses a
different PN number. AES-GCM-128 is the default cipher suite. In default mode, key size is 128 bits. In AES-GCM-256 mode, key size is 256 bits. The basic AES core processes 16 bytes of data at a time. For the 128-bit key, it takes 12 clocks to complete processing of 128 bits data. For 256-bit key, it takes 16 clocks to complete processing of one 128-bit data.

##### 10.7.2.1.3 MAC Tx
In MGBE wrapper, this block interfaces with PCS through the XGMII interface for MGBE. This block converts internal information like SOP, EOP, data, and so on to XGMII data and control characters. In ETHER_QOS wrapper, this block converts internal information like SOP, EOP, and data to RGMII data and control characters.
- FCS (Ethernet CRC) RECALCULATION
When the MACsec header and ICV are inserted into the packet, FCS is recalculated. For bypass packets with modifications, FCS is regenerated. In short, in the MACsec IP, FCS is always calculated for both, bypassed, and not-bypassed packets.

##### 10.7.2.3.2 Processing of Preemptable Packets
As shown in the preceding diagram, intermediate preemptable fragment packets do not have the Ethernet header. MACsec does not have support for complete store and forward of the packets. Each fragment of the packet is transmitted out immediately after being processed. That means, to support processing of the preemptable packets, the MACsec Engine must store the following context information of the preemptable packet in its internal registers.
- SAK
- HKEY
Intermediate HASH output from GHASH Computation.
- Unused bytes from the previous fragment (These are used along with the new bytes of the
next fragment for GHASH computation). When the next fragment of preemptable packet is received, hardware restores the context and resumes the processing of the preemptable packet. The processing of preemptable packets by MACsec Engine is similar in both transmit and receive directions. Note: Fragments of a preempted frame are required to arrive in sequential order at MACSEC. This should not be a problem since preemption is only applicable on point-to-point links. But if fragments are not received sequentially (there can be Express frames in the middle) at MACSEC, MACSEC corrupts mCRC (or CRC if it is final fragment) of the fragment before forwarding it. This is true in both Transmit and Receive direction. It is assumed that receiving MAC discards the frame because of wrong mCRC or CRC.

- MACsec Functional Description
- Replay protection with Preemptable packets
In the receive direction, the Packet Number (PN) present in SecTAG field of MACsec packet is used to provide replay protection. If MACsec is a point-to-point feature, then there is no scope for out of order packets. Strict Replay protection can be enabled irrespective of whether packets are preemptable or express packets.
- But if MACsec is used as an end-to-end feature, then the frames belonging to same SC might
arrive out of order because of preemption support in downstream devices. In this case, there are two options to carry out replay protection. Increase Replay window size.
- Use two Secure channels. One for Preemptable traffic and one for Express traffic. To
support this feature, the SC LUT has a field called “Preempt pkt”. Based on this bit, software can set up different SC channel for Express packets and non-Express packets. If software wants to set up same SC channel for both Express and Preemptable packets, then this bit should be masked out.
#### 10.7.2.4 MTU Check on Egress and Ingress Packets
In egress direction, the length of the packet (post-MACsec processing) is compared to TX MTU register (MACSEC_TX_MTU_LEN_0). If length is greater than MTU value, then CRC of outgoing packet is corrupted. Similarly, in Ingress direction, the length of the received packet from line, is compared to RX MTU register (MACSEC_RX_MTU_LEN_0). If length is greater than MTU value, then CRC of packet going to MAC is corrupted. Note: MTU check ignores (MAC-DA, MAC-SA, FCS) fields in the packet length.

##### 10.7.2.3.2 Processing of Preemptable Packets
As shown in the preceding diagram, intermediate preemptable fragment packets do not have the Ethernet header. MACsec does not have support for complete store and forward of the packets. Each fragment of the packet is transmitted out immediately after being processed. That means, to support processing of the preemptable packets, the MACsec Engine must store the following context information of the preemptable packet in its internal registers.
- SAK
- HKEY
Intermediate HASH output from GHASH Computation.
- Unused bytes from the previous fragment (These are used along with the new bytes of the
next fragment for GHASH computation). When the next fragment of preemptable packet is received, hardware restores the context and resumes the processing of the preemptable packet. The processing of preemptable packets by MACsec Engine is similar in both transmit and receive directions. Note: Fragments of a preempted frame are required to arrive in sequential order at MACSEC. This should not be a problem since preemption is only applicable on point-to-point links. But if fragments are not received sequentially (there can be Express frames in the middle) at MACSEC, MACSEC corrupts mCRC (or CRC if it is final fragment) of the fragment before forwarding it. This is true in both Transmit and Receive direction. It is assumed that receiving MAC discards the frame because of wrong mCRC or CRC.

- MACsec Functional Description
- Replay protection with Preemptable packets
In the receive direction, the Packet Number (PN) present in SecTAG field of MACsec packet is used to provide replay protection. If MACsec is a point-to-point feature, then there is no scope for out of order packets. Strict Replay protection can be enabled irrespective of whether packets are preemptable or express packets.
- But if MACsec is used as an end-to-end feature, then the frames belonging to same SC might
arrive out of order because of preemption support in downstream devices. In this case, there are two options to carry out replay protection. Increase Replay window size.
- Use two Secure channels. One for Preemptable traffic and one for Express traffic. To
support this feature, the SC LUT has a field called “Preempt pkt”. Based on this bit, software can set up different SC channel for Express packets and non-Express packets. If software wants to set up same SC channel for both Express and Preemptable packets, then this bit should be masked out.
#### 10.7.2.4 MTU Check on Egress and Ingress Packets
In egress direction, the length of the packet (post-MACsec processing) is compared to TX MTU register (MACSEC_TX_MTU_LEN_0). If length is greater than MTU value, then CRC of outgoing packet is corrupted. Similarly, in Ingress direction, the length of the received packet from line, is compared to RX MTU register (MACSEC_RX_MTU_LEN_0). If length is greater than MTU value, then CRC of packet going to MAC is corrupted. Note: MTU check ignores (MAC-DA, MAC-SA, FCS) fields in the packet length.
