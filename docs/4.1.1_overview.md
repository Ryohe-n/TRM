# 4.1.1 Overview

@{Key=4.1.1; Title=Overview; Body=involves a mostly platform agnostic view of the boot process, starting from the point that the BPMP Cortex®-R5 is brought out of reset, to the point that the operating system software starts executing on the CPU.
#### 4.1.1.1 Boot Code Terms
- Primary Boot Code; also referred to as the Boot ROM, PSCROM, MCE-MROM
- There are three primary ROMs. Boot ROM is on-chip IROM in the BPMP cluster and
executed by the BPMP_R5; PSCROM is on-chip IROM in Platform Security Controller (PSC) cluster executed by PSC_RISCV; and MROM is on-chip IROM in CPU Complex (CCPLEX) cluster executed by a Micro-Code Engine (MCE).
- Secondary Boot Code
- The code that executes immediately after Primary Boot Code is referred to as the
Secondary Boot Code. The code is stored in external non-volatile memory (NVM), this is called the Secondary Boot
- Media and code is fetched by the Boot ROM into volatile chip memory. PSCROM validates
the secondary boot code before turning over execution to it.
- Depending on the platform, the security requirements, the software ownership, and/or other
software considerations, each subsequent piece of Boot Code may reside on the same boot media or different boot media, and could be a chain of fetch/execute sequences. These may be fetched from the boot media into either on-chip RAM or into DRAM. These definitions, used in this document and other hardware architecture documentation, are generic and are platform agnostic for boot components. However, due to legacy and diversity in platform software implementations, these may map to different terms in different platform software.

#### 4.1.1.2 Boot Flow and Branches
##### 4.1.1.2.1 Boot Types
This subsection briefly introduces the various boot types supported in the SoC. More sequencing details are covered in later sections.
- Frozen Boot – Both the application processor and PMIC, or platform power and reset
components, are power cycled. PMIC initializes itself and then supplies power and resets the system. Typically, this occurs when removing the battery on mobile systems, or cold start automotive systems. For BPMP and PSC, it is identical to a cold boot. The differences are only in the additional external PMIC startup steps. Cold Boot – This is subset of frozen boot. Application processor receives an external main or internally generated reset assertion prior to this boot. Boot ROM loads the BCT and
- Secondary Boot Code from the Secondary Boot Media. A Cold Boot can be initiated by
external L0 or internally generated resets. Warm Boot – This refers to the SC7-exit sequence. In this boot type, recovery is from a state where some context (state) is stored in the Always On (AO) logic resources and DRAM retains state. Almost all of the chip is powered off and loses state, including devices, and the MC configuration registers. BPMP and PSC restore this state from AO resources while booting up during an SC7 exit. The DRAM devices are put in self-refresh before SC7 entry and thus retain state. A large portion of the SC7 restore code itself is loaded from DRAM.
- Note: Sequences for exits from other system power states shallower than SC7 do not
originate in Boot ROM and do not involve BPMP boot processes. For this reason, these are not discussed in the boot context. However, some of these state exits may involve the SoC and require a micro-architectural boot of Microcode.
#### 4.1.1.3 Boot Code Stages
##### 4.1.1.3.1 Platform Agnostic Definitions
This section defines various boot stages of execution. This chapter contains a very generic introduction to the boot stages to establish context. More accurate implementation of the specific boot architecture is presented in later chapters. Boot ROM: All platforms use the Boot ROM in the boot process. It is the very first code that the BPMP executes upon coming out of reset (including reset during SC7 exit) during a system boot. It is contained in an on-chip ROM and executes completely on the BPMP. The Boot ROM represents the root of the Chain-of-Trust (CoT) or Root-of-Trust (RoT) for boot. PSCROM: All platforms use the PSC ROM in the boot process. It is the very first code that the PSC_RISCV executes upon coming out of reset (including reset during SC7 exit) during a system

- Boot Process
boot. It is contained in an on-chip ROM and executes completely on the PSC_RISCV. The PSCROM represents the root of the Chain-of-Trust (CoT) or Root-of-Trust (RoT) for boot.
- Microboot-Stage1 (MB1): MB1 is a boot image loaded by the Boot ROM from a secondary boot
storage (typically flash) device. This code is authenticated and decrypted by PSC code loaded from the PSC ROM before execution. MB1 is loaded into on-chip RAM and authenticated in and executed from that RAM. It typically contains the DRAM initialization routine before loading the next boot stage which is executed on the BPMP. MB1 is functionally treated as an extension of the BPMP Boot ROM.
- Microboot-Stage2 (MB2): MB1 is followed by another boot stage – MB2, which is optionally
encrypted and rollback-protected and signed by the OEM-authority, and is executed on BPMP. NVIDIA provides binary for MB2 to the OEM. PSC-BootLoader (PSC-BL): PSC-BL is a boot image loaded by the Boot ROM from a secondary boot storage (typically flash) device. This code is authenticated and decrypted by PSCROM before execution. PSC-BL is loaded into on-chip RAM and authenticated in and executed from that RAM. It typically contains the secure boot code for validating and auditing the next boot stage which is executed on the PSC. PSC-BL is functionally treated as an extension of the PSC Boot ROM.
- Power Management Firmware (BPMP-FW): BPMP-FW is a boot image loaded by the MB1 from a
secondary boot storage (typically flash) device. This code is authenticated and decrypted by PSC-
- BL before execution. BPMP-FW is loaded and authenticated in and executed from that
BPMP. During runtime BPMP-FW services clock, reset and power management requests from other CPUs which are executed on the BPMP. ATF or monitor and Trusted OS or Secure OS: The main CPU (Arm® Cortex-A78) is always brought out of reset in trusted mode, so it can only run TrustZone code. Trusted OS (called TZOS, TOS, or Secure Monitor) and ATF are loaded by CCPLEX_MB2. This TOS executes, and then branches to the (pre-authenticated) Boot Loader (BL-CPU) that runs on the CPU, before passing control to the non- TrustZone OS code. Boot Loader: This stage is loaded by CCPLEX_MB2 from the secondary boot media into DRAM. This stage does not execute out of on-chip RAM. It is typically significantly larger than the previous stages, so the Microboot may configure the boot media to its highest speed possible before loading the Boot Loader. After this stage the software stack on a given platform may vary, for example elect to have a Hypervisor layer or not, a small kernel or a rich-kernel, and so on. These decisions depend on platform software requirements and are beyond the scope of this document. Hypervisor (EL2): HV (VMM) is loaded by CPU_BL to provide virtualization in the software stack. It loads different guests on the Virtual-CPU by either restricting access to a given device by a guest VM or managing the device on guest’s behalf. Note: To maintain the Chain-of-trust at any stage of boot execution above, the previous boot stage shall authenticate the next stage, before passing execution control to it.

##### 4.1.1.3.4 L0 Reset Boot
This is the most primal form of boot, where all states in the system come out of reset, and the PMIC power cycles all of the chip. When not otherwise mentioned, it is this boot form that is described in this document. Taking this particular sequence as reference, the rest of the boot flows are derived. There are some exceptions to this, in particular the warm boot sequence. Since this is a recovery of the system from a previous known state stored in DRAM, and so the boot sequence and pieces of software executing are decidedly different.
##### 4.1.1.3.5 SC7 Exit
SC7 exit, is the boot type that is the only warm boot exit in this list. SC7 is a "suspend to RAM" low-power state of the SoC in which DRAM content is preserved. AON content is preserved (which implies that the AON rail is ON), and SoC/CPU/GPU rails are powered off. An SC7 exit is a warm boot and follows a different boot flow where SoC units are booted from power-on, but context from DRAM and AON is 'restored' as opposed to being read from flash storage. Boot devices are not used in SC7 exit.

- Boot Process
##### 4.1.1.3.6 WDT 5th Expiry (L1 Resets)
These are L1 resets whose boot sequence is identical to a cold reset. Some very exclusive logic, only on L0 and the AON scratch RAM, is not reset in this boot type. Software differentiates these boot types and takes specific actions based on the AON scratch register information.
##### 4.1.1.3.7 WDT 4th Expiry, HSM (L2 Resets)
This is the only L2 reset in the system. The system is rebooted to stream the debug state to an external host after an unexpected event caused WDT reset. A HSM software initiated reset can also cause an L2 reset.
##### 4.1.1.3.8 Key-off/Key-on IST Boot
- In System Test (IST) refers to executing structural memory and scan tests in the field for
automotive platforms. It is targeted to provide coverage for latent faults towards meeting the ISO 26262 ASIL requirements. IST is intended to be executed at key-on or key-off of the vehicle. From boot point of view, Key-on/Key-off IST Boot flow is defined which primarily diverges from cold boot flow at MB1/PSC-BL.
##### 4.1.1.3.9 DRAM Page Blacklisting Boot Flow
- Safety requirements sometimes necessitate the need to detect and/or pre-empt DRAM page
errors. The SoC supports memory ECC, which corrects single-bit errors and detects double-bit errors. Some of the related acronyms to address these include single-bit error correction (SEC) and double-bit error detection (DED).
- Error Detection
In a live system, a DRAM error can occur at any time, i.e., radiation event, gamma ray, etc. This could be transient or permanent, single-bit error (SBE) or double-bit error (DBE). On either an accrued consistent SBE cluster or any DBE error, software logs the cause of reset and the bad DRAM page location in AO scratch, then triggers a cold reset through the HSM.
- Blacklisting
On the next cold boot (identified as an HSM reset cause), MB1 reads the cause, determines the bad page, then does a test to make sure it was a permanent error page. It then takes a slightly different boot path involving storage of this bad page to the flash device (NVM) to accrue all similar bad pages that may have been detected over the lifetime of the product. It then triggers a regular cold boot.

- Boot Process
- Accounting for Blacklisted Pages
On every cold boot path, MB1 always reads the flash for bad pages. While allocating physically mapped carveouts, MB1 reassigns carveouts to avoid bad pages. This bad page information is also passed to downstream boot stages, where the VA -> PA mappings makes sure that bad pages are never allocated. The granularity is in 4k physical pages. On a warm boot, all DRAM context and addresses are restored as is, so warm boot is agnostic to blacklisting. It is a cold boot only phenomenon.
- Other Clarifications
Blacklisting is only for DRAM, not for SysRAM.
##### 4.1.1.3.10 RCM Flashing
- RCM flashing boot is a boot path where the Boot ROM goes into RCM (recovery mode) and does
not boot up the system functionally. Instead, after performing primary security functions, it initializes the XUSB device in device mode and lets a remote host load an applet for the BPMP to execute. This allows the chip to flash the non-volatile boot devices with functional boot images from the host for the next boot.
##### 4.1.1.3.11 RCM Debug Boot
RCM debug boot is a debug boot path where the cold boot path is experimented upon by using the boot images from the host to load directly into DRAM. This skips the actual flashing process (to save development time) into devices for development boot images. Early Boot ROM sequencing is identical to RCM flashing. The flow is like RCM, but the booting sequence is functional for testing purposes. Typically, images are loaded to DRAM, and BPMP executes developmental boot code from the loaded binaries. From the Boot ROM’s perspective, it is identical to RCM flashing, but MB1 onwards the boot flow is different.
##### 4.1.1.3.12 Boot Chain Support
Some applications require a backup boot partition to prevent bricking or hanging the system due to a bad flashing image update. These may use this boot flow to flash the primary partition instead of using RCM mode. This is primarily for runtime firmware updates. This is also in line with always having at least one safe reliable set of boot images.
##### 4.1.1.3.13 FSKP Boot
Factory Secure Key Provisioning (FSKP) is a special mode used in production mode (production mode fuse burnt, but odm_production mode fuse not burnt) to write/burn their OEM fuse keys in the chip. They can optionally use production cold boot flow, or mimic RCM boot to achieve this.

##### 4.1.1.3.14 FA Mode Boot
- FA mode is supported in early Boot ROM, which allows an early exit out of Boot ROM without
actually going through functional boot much like RCM mode. If the FA fuse is sampled burnt by Boot ROM, it shall do an early secure exit, and wait for UART commands.
#### 4.1.1.4 Boot Devices (Boot Sources)
This section describes secondary boot devices, i.e., the NVM interfaces that store the next level of instructions and data to carry on the boot process after the Boot ROM has started executing. The Boot ROM has code that (depending on fuses and straps) directs its fetch to the right boot source interface.
##### 4.1.1.4.1 eMMC
The SDMMC Controller is capable of interfacing to Secure Digital (SD) / Embedded SD (eSD), SD I/O (SDIO), and eMMC cards. It has a direct MCCIF (AXICIF) interface and is capable of initiating data transfers between memory and external card. It has an APB Slave interface to access its configuration registers. The SDMMC controller supports two different bus protocols: SD/SDIO bus protocol for the SD/SDIO cards and eMMC bus protocol for eMMC cards.
- Only eMMC is the supported secondary boot media and only SDMMC4 controller must be used to
access eMMC boot device by Boot ROM.

##### 4.1.1.4.2 QSPI
The QSPI Controller is a Serial Communications link controller between the application processor and Serial NOR flash devices. QSPI Controller supports independent transmit and receive FIFOs of 64x 32 bits each for PIO mode and a FIFO of 40x 64 bits for DMA mode. Since QSPI has only half- duplex use-case, single FIFO is used for Tx and Rx in DMA mode. Software can program the controller to generate transactions of required packet length on the bus, where a transaction is a sequence of packets in either direction. Software can manually read/write the FIFOs in PIO mode.
- QSPI PIO mode of operation can also use GPC_DMA to read and write from the independent
transmit/receive FIFOs as required. QSPI DMA mode of operation uses native DMA engine to read and write from the FIFO to memory as required.

## 4.2 Power Management Controller (PMC)}(@{Key=4.1.1; Title=Overview; Body=involves a mostly platform agnostic view of the boot process, starting from the point that the BPMP Cortex®-R5 is brought out of reset, to the point that the operating system software starts executing on the CPU.
#### 4.1.1.1 Boot Code Terms
- Primary Boot Code; also referred to as the Boot ROM, PSCROM, MCE-MROM
- There are three primary ROMs. Boot ROM is on-chip IROM in the BPMP cluster and
executed by the BPMP_R5; PSCROM is on-chip IROM in Platform Security Controller (PSC) cluster executed by PSC_RISCV; and MROM is on-chip IROM in CPU Complex (CCPLEX) cluster executed by a Micro-Code Engine (MCE).
- Secondary Boot Code
- The code that executes immediately after Primary Boot Code is referred to as the
Secondary Boot Code. The code is stored in external non-volatile memory (NVM), this is called the Secondary Boot
- Media and code is fetched by the Boot ROM into volatile chip memory. PSCROM validates
the secondary boot code before turning over execution to it.
- Depending on the platform, the security requirements, the software ownership, and/or other
software considerations, each subsequent piece of Boot Code may reside on the same boot media or different boot media, and could be a chain of fetch/execute sequences. These may be fetched from the boot media into either on-chip RAM or into DRAM. These definitions, used in this document and other hardware architecture documentation, are generic and are platform agnostic for boot components. However, due to legacy and diversity in platform software implementations, these may map to different terms in different platform software.

#### 4.1.1.2 Boot Flow and Branches
##### 4.1.1.2.1 Boot Types
This subsection briefly introduces the various boot types supported in the SoC. More sequencing details are covered in later sections.
- Frozen Boot – Both the application processor and PMIC, or platform power and reset
components, are power cycled. PMIC initializes itself and then supplies power and resets the system. Typically, this occurs when removing the battery on mobile systems, or cold start automotive systems. For BPMP and PSC, it is identical to a cold boot. The differences are only in the additional external PMIC startup steps. Cold Boot – This is subset of frozen boot. Application processor receives an external main or internally generated reset assertion prior to this boot. Boot ROM loads the BCT and
- Secondary Boot Code from the Secondary Boot Media. A Cold Boot can be initiated by
external L0 or internally generated resets. Warm Boot – This refers to the SC7-exit sequence. In this boot type, recovery is from a state where some context (state) is stored in the Always On (AO) logic resources and DRAM retains state. Almost all of the chip is powered off and loses state, including devices, and the MC configuration registers. BPMP and PSC restore this state from AO resources while booting up during an SC7 exit. The DRAM devices are put in self-refresh before SC7 entry and thus retain state. A large portion of the SC7 restore code itself is loaded from DRAM.
- Note: Sequences for exits from other system power states shallower than SC7 do not
originate in Boot ROM and do not involve BPMP boot processes. For this reason, these are not discussed in the boot context. However, some of these state exits may involve the SoC and require a micro-architectural boot of Microcode.
#### 4.1.1.3 Boot Code Stages
##### 4.1.1.3.1 Platform Agnostic Definitions
This section defines various boot stages of execution. This chapter contains a very generic introduction to the boot stages to establish context. More accurate implementation of the specific boot architecture is presented in later chapters. Boot ROM: All platforms use the Boot ROM in the boot process. It is the very first code that the BPMP executes upon coming out of reset (including reset during SC7 exit) during a system boot. It is contained in an on-chip ROM and executes completely on the BPMP. The Boot ROM represents the root of the Chain-of-Trust (CoT) or Root-of-Trust (RoT) for boot. PSCROM: All platforms use the PSC ROM in the boot process. It is the very first code that the PSC_RISCV executes upon coming out of reset (including reset during SC7 exit) during a system

- Boot Process
boot. It is contained in an on-chip ROM and executes completely on the PSC_RISCV. The PSCROM represents the root of the Chain-of-Trust (CoT) or Root-of-Trust (RoT) for boot.
- Microboot-Stage1 (MB1): MB1 is a boot image loaded by the Boot ROM from a secondary boot
storage (typically flash) device. This code is authenticated and decrypted by PSC code loaded from the PSC ROM before execution. MB1 is loaded into on-chip RAM and authenticated in and executed from that RAM. It typically contains the DRAM initialization routine before loading the next boot stage which is executed on the BPMP. MB1 is functionally treated as an extension of the BPMP Boot ROM.
- Microboot-Stage2 (MB2): MB1 is followed by another boot stage – MB2, which is optionally
encrypted and rollback-protected and signed by the OEM-authority, and is executed on BPMP. NVIDIA provides binary for MB2 to the OEM. PSC-BootLoader (PSC-BL): PSC-BL is a boot image loaded by the Boot ROM from a secondary boot storage (typically flash) device. This code is authenticated and decrypted by PSCROM before execution. PSC-BL is loaded into on-chip RAM and authenticated in and executed from that RAM. It typically contains the secure boot code for validating and auditing the next boot stage which is executed on the PSC. PSC-BL is functionally treated as an extension of the PSC Boot ROM.
- Power Management Firmware (BPMP-FW): BPMP-FW is a boot image loaded by the MB1 from a
secondary boot storage (typically flash) device. This code is authenticated and decrypted by PSC-
- BL before execution. BPMP-FW is loaded and authenticated in and executed from that
BPMP. During runtime BPMP-FW services clock, reset and power management requests from other CPUs which are executed on the BPMP. ATF or monitor and Trusted OS or Secure OS: The main CPU (Arm® Cortex-A78) is always brought out of reset in trusted mode, so it can only run TrustZone code. Trusted OS (called TZOS, TOS, or Secure Monitor) and ATF are loaded by CCPLEX_MB2. This TOS executes, and then branches to the (pre-authenticated) Boot Loader (BL-CPU) that runs on the CPU, before passing control to the non- TrustZone OS code. Boot Loader: This stage is loaded by CCPLEX_MB2 from the secondary boot media into DRAM. This stage does not execute out of on-chip RAM. It is typically significantly larger than the previous stages, so the Microboot may configure the boot media to its highest speed possible before loading the Boot Loader. After this stage the software stack on a given platform may vary, for example elect to have a Hypervisor layer or not, a small kernel or a rich-kernel, and so on. These decisions depend on platform software requirements and are beyond the scope of this document. Hypervisor (EL2): HV (VMM) is loaded by CPU_BL to provide virtualization in the software stack. It loads different guests on the Virtual-CPU by either restricting access to a given device by a guest VM or managing the device on guest’s behalf. Note: To maintain the Chain-of-trust at any stage of boot execution above, the previous boot stage shall authenticate the next stage, before passing execution control to it.

##### 4.1.1.3.4 L0 Reset Boot
This is the most primal form of boot, where all states in the system come out of reset, and the PMIC power cycles all of the chip. When not otherwise mentioned, it is this boot form that is described in this document. Taking this particular sequence as reference, the rest of the boot flows are derived. There are some exceptions to this, in particular the warm boot sequence. Since this is a recovery of the system from a previous known state stored in DRAM, and so the boot sequence and pieces of software executing are decidedly different.
##### 4.1.1.3.5 SC7 Exit
SC7 exit, is the boot type that is the only warm boot exit in this list. SC7 is a "suspend to RAM" low-power state of the SoC in which DRAM content is preserved. AON content is preserved (which implies that the AON rail is ON), and SoC/CPU/GPU rails are powered off. An SC7 exit is a warm boot and follows a different boot flow where SoC units are booted from power-on, but context from DRAM and AON is 'restored' as opposed to being read from flash storage. Boot devices are not used in SC7 exit.

- Boot Process
##### 4.1.1.3.6 WDT 5th Expiry (L1 Resets)
These are L1 resets whose boot sequence is identical to a cold reset. Some very exclusive logic, only on L0 and the AON scratch RAM, is not reset in this boot type. Software differentiates these boot types and takes specific actions based on the AON scratch register information.
##### 4.1.1.3.7 WDT 4th Expiry, HSM (L2 Resets)
This is the only L2 reset in the system. The system is rebooted to stream the debug state to an external host after an unexpected event caused WDT reset. A HSM software initiated reset can also cause an L2 reset.
##### 4.1.1.3.8 Key-off/Key-on IST Boot
- In System Test (IST) refers to executing structural memory and scan tests in the field for
automotive platforms. It is targeted to provide coverage for latent faults towards meeting the ISO 26262 ASIL requirements. IST is intended to be executed at key-on or key-off of the vehicle. From boot point of view, Key-on/Key-off IST Boot flow is defined which primarily diverges from cold boot flow at MB1/PSC-BL.
##### 4.1.1.3.9 DRAM Page Blacklisting Boot Flow
- Safety requirements sometimes necessitate the need to detect and/or pre-empt DRAM page
errors. The SoC supports memory ECC, which corrects single-bit errors and detects double-bit errors. Some of the related acronyms to address these include single-bit error correction (SEC) and double-bit error detection (DED).
- Error Detection
In a live system, a DRAM error can occur at any time, i.e., radiation event, gamma ray, etc. This could be transient or permanent, single-bit error (SBE) or double-bit error (DBE). On either an accrued consistent SBE cluster or any DBE error, software logs the cause of reset and the bad DRAM page location in AO scratch, then triggers a cold reset through the HSM.
- Blacklisting
On the next cold boot (identified as an HSM reset cause), MB1 reads the cause, determines the bad page, then does a test to make sure it was a permanent error page. It then takes a slightly different boot path involving storage of this bad page to the flash device (NVM) to accrue all similar bad pages that may have been detected over the lifetime of the product. It then triggers a regular cold boot.

- Boot Process
- Accounting for Blacklisted Pages
On every cold boot path, MB1 always reads the flash for bad pages. While allocating physically mapped carveouts, MB1 reassigns carveouts to avoid bad pages. This bad page information is also passed to downstream boot stages, where the VA -> PA mappings makes sure that bad pages are never allocated. The granularity is in 4k physical pages. On a warm boot, all DRAM context and addresses are restored as is, so warm boot is agnostic to blacklisting. It is a cold boot only phenomenon.
- Other Clarifications
Blacklisting is only for DRAM, not for SysRAM.
##### 4.1.1.3.10 RCM Flashing
- RCM flashing boot is a boot path where the Boot ROM goes into RCM (recovery mode) and does
not boot up the system functionally. Instead, after performing primary security functions, it initializes the XUSB device in device mode and lets a remote host load an applet for the BPMP to execute. This allows the chip to flash the non-volatile boot devices with functional boot images from the host for the next boot.
##### 4.1.1.3.11 RCM Debug Boot
RCM debug boot is a debug boot path where the cold boot path is experimented upon by using the boot images from the host to load directly into DRAM. This skips the actual flashing process (to save development time) into devices for development boot images. Early Boot ROM sequencing is identical to RCM flashing. The flow is like RCM, but the booting sequence is functional for testing purposes. Typically, images are loaded to DRAM, and BPMP executes developmental boot code from the loaded binaries. From the Boot ROM’s perspective, it is identical to RCM flashing, but MB1 onwards the boot flow is different.
##### 4.1.1.3.12 Boot Chain Support
Some applications require a backup boot partition to prevent bricking or hanging the system due to a bad flashing image update. These may use this boot flow to flash the primary partition instead of using RCM mode. This is primarily for runtime firmware updates. This is also in line with always having at least one safe reliable set of boot images.
##### 4.1.1.3.13 FSKP Boot
Factory Secure Key Provisioning (FSKP) is a special mode used in production mode (production mode fuse burnt, but odm_production mode fuse not burnt) to write/burn their OEM fuse keys in the chip. They can optionally use production cold boot flow, or mimic RCM boot to achieve this.

##### 4.1.1.3.14 FA Mode Boot
- FA mode is supported in early Boot ROM, which allows an early exit out of Boot ROM without
actually going through functional boot much like RCM mode. If the FA fuse is sampled burnt by Boot ROM, it shall do an early secure exit, and wait for UART commands.

##### 4.1.1.4.2 QSPI
The QSPI Controller is a Serial Communications link controller between the application processor and Serial NOR flash devices. Software can program the controller to generate transactions of required packet length on the bus, where a transaction is a sequence of packets in either direction. Software can manually read/write the FIFOs in PIO mode.
