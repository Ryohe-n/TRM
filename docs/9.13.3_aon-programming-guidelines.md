##### 9.13.3.1.1 Cluster Power State Definitions
The AO Power State framework consists of SPE power states on one hand and the cluster's activity levels mapped to System Power States on the other. The SPE subsystem in the AO cluster is in one of four power states: 1. 2. 3. 4.
- SPE-Active
- SPE-Idle
- SPE-Standby
- SPE-Dormant

- AON Programming Guidelines
**Figure 9.101 SPE Power States Hierarchy within System Power States Framework**
These power states are defined by the sensor processor and sensor I/Os expected activity levels. The system and power management functions in the AO cluster operate at various levels of activity but are never switched off completely except when the SoC is fully powered off (shut down). At the cluster level there is no specific idle or standby state defined. The AO cluster is an always-on domain in the chip. In any of the SoC power states, the AO cluster could be in any of its own power states. However, there are some constraints on the AO cluster state while the chip transitions into and out of the SC7 SoC states.

- AON Programming Guidelines
**Figure 9.102 AO Rail Power States Transition Framework**
In the steady state, any of the AO cluster power states (except complete power-off) are supported by hardware in any of the SoC power states.
##### 9.13.3.1.2 Cortex-R5 CPU Halted (Clock Gating)
If the CPU is idle for a short period of time during WFI state, then it can be halted which results in its CPU clock-gating. The WFI instruction is executed for putting CPU into halt. After executing WFI, CPU disables most of the clocks in the CPU except the clock for the small wake up logic clock. Exit from halt state occurs when the CPU detects an interrupt on IRQ or FIQ pins.
##### 9.13.3.1.3 TCM Clock Gating
The TCM interface clock can be gated when there are no pending DMA requests and the Cortex-R5 is halted. The nCLKSTOPPED signal is used to indicate when the TCM clock can be gated. The TCM clock must be restarted in no more than four clocks after nCLKSTOPPED is deasserted.
##### 9.13.3.1.4 OSC Power Down
To reduce SC7 power when the AON Cluster is idle for an extended period (greater than 3 ms), all AON Cluster clocks except for the 32 kHz clock can be disabled by powering down the OSC pad. When the OSC is powered down, only the RTC, or an external event, can wake the AON Cluster.

#### 9.13.3.2 AON Cluster SC7 Entry
Access from the AON cluster to the SOC is blocked by the software when entering SC7. If requests are in flight when the block logic is engaged it is possible a deadlock could occur. Software should follow the sequence below to ensure this deadlock does not happen. 1. 2.
- BPMP notifies AON FW to stop all SOC access (MMIO and DRAM) via an IPC message
- AON firmware performs all necessary steps to be able to operate with no SOC access. (Swap
code into TCM, flush, and Disable caches, etc.) After this point AON firmware is executing only from TCM.
- AON firmware sends IPC to BPMP that it is safe to disable SOC access
BPMP acknowledges IPC by writing to HSP mailbox in AON. AON FW detects AON mailbox write and acknowledges by writing to AON HSP.
- BPMP polls AON HSP waiting for the acknowledgment from AON. When the acknowledgment
is received, PSC writes to AOPM to enable the block function. 3. 4. 5. 6. 7.
#### 9.13.3.3 AON Cortex-R5 Activity Monitor
The unit activity monitor is a part of the AO-PM module. The interrupt from the unit ACTMON is not merged with the AO-PM interrupt. It is separately routed to the local VIC and to LIC.
- AON uses a type-1 unit ACTMON [refer to the Activity Monitor (ACTMON) chapter of this
document] to detect idle periods of the processor complex and monitor a few key events that the Cortex-R5 exposes on its event bus. The idle state of the Cortex-R5 is not exposed to the event bus. Therefore, the idle signals nCLKSTOPPED, nWFIPIPESTOPPED, and nWFEPIPESTOPPED must be separately exported to detect idle period statistics and use the information for scaling the performance of the Cortex-R5. The outputs from the Cortex-R5 are sampled at the processor clock frequency and a cumulative counter incremented for every cycle that the unit_active signal is asserted. The cumulative count is

- AON Programming Guidelines
compared with a target count. When the cumulative count in processor clock rate reaches the target value, the ACTMON is notified. The EVENTBUS signals hooked up to the ACTMON are indicative of the following events.
- EVENTBUS[5]: Indicates that the instruction is architecturally executed. Asserted on every
cycle when an instruction was architecturally executed by the processor.
- EVENTBUS[16]: Stalls because instruction buffer cannot deliver an instruction. This can
indicate an instruction cache miss. Asserted every cycle where the condition is present.
- EVENTBUS[19]: External memory request. Examples of this are cache refill, non-cacheable
accesses, write-through writes, cache line evictions (write-back).
- EVENTBUS[20]: Stalls because of Load Store Unit (LSU) being busy. The event takes place
each clock cycle where the condition is met. A high incidence of this event indicates the pipeline is often waiting for transactions to complete on the external bus. The unit ACTMON uses aon_cpu_clk as the refclk.

#### 9.13.3.2 AON Cluster SC7 Entry
Access from the AON cluster to the SOC is blocked by the software when entering SC7. If requests are in flight when the block logic is engaged it is possible a deadlock could occur. Software should follow the sequence below to ensure this deadlock does not happen. 1. 2.
- BPMP notifies AON FW to stop all SOC access (MMIO and DRAM) via an IPC message
- AON firmware performs all necessary steps to be able to operate with no SOC access. (Swap
code into TCM, flush, and Disable caches, etc.) After this point AON firmware is executing only from TCM.
- AON firmware sends IPC to BPMP that it is safe to disable SOC access
BPMP acknowledges IPC by writing to HSP mailbox in AON. AON FW detects AON mailbox write and acknowledges by writing to AON HSP.
- BPMP polls AON HSP waiting for the acknowledgment from AON. When the acknowledgment
is received, PSC writes to AOPM to enable the block function. 3. 4. 5. 6. 7.
#### 9.13.3.3 AON Cortex-R5 Activity Monitor
The unit activity monitor is a part of the AO-PM module. The interrupt from the unit ACTMON is not merged with the AO-PM interrupt. It is separately routed to the local VIC and to LIC.
- AON uses a type-1 unit ACTMON [refer to the Activity Monitor (ACTMON) chapter of this
document] to detect idle periods of the processor complex and monitor a few key events that the Cortex-R5 exposes on its event bus. The idle state of the Cortex-R5 is not exposed to the event bus. Therefore, the idle signals nCLKSTOPPED, nWFIPIPESTOPPED, and nWFEPIPESTOPPED must be separately exported to detect idle period statistics and use the information for scaling the performance of the Cortex-R5. The outputs from the Cortex-R5 are sampled at the processor clock frequency and a cumulative counter incremented for every cycle that the unit_active signal is asserted. The cumulative count is

compared with a target count. When the cumulative count in processor clock rate reaches the target value, the ACTMON is notified. The EVENTBUS signals hooked up to the ACTMON are indicative of the following events.
- EVENTBUS[5]: Indicates that the instruction is architecturally executed. Asserted on every
cycle when an instruction was architecturally executed by the processor.
- EVENTBUS[16]: Stalls because instruction buffer cannot deliver an instruction. This can
indicate an instruction cache miss. Asserted every cycle where the condition is present.
- EVENTBUS[19]: External memory request. Examples of this are cache refill, non-cacheable

- EVENTBUS[20]: Stalls because of Load Store Unit (LSU) being busy. The event takes place
each clock cycle where the condition is met. A high incidence of this event indicates the pipeline is often waiting for transactions to complete on the external bus. The unit ACTMON uses aon_cpu_clk as the refclk.
