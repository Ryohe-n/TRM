# 4.2.3 PMC Programming Guidelines

#### 4.2.3.1 SC7 Hardware Sequencer
This entire sequencer is in 32 kHz clock domain. The hardware sequencer is triggered when the
- BPMP software writes the DPD_ENABLE register after specifying the target power state
[PMC_IMPL_SC7_CONFIG_0_SC7_TARGET= SC7].
- The BPMP software excludes the AO I/Os used in SC7 state by setting up the
- PMC_IMPL_IO_SYS_DPD_OFF_MASK_0, PMC_IMPL_IO_AO_HV_DPD_OFF_MASK_0 and
- PMC_IMPL_IO_AO_DPD_OFF_MASK_0 registers before triggering hardware actions. This way the
required pads for SPE operation while in SC7 they are excluded from entering DPD.

**Figure 4.31 SC7 Hardware Sequencer**
The first state in the sequencer i.e., SC7_ASSERT_SLEEP exists in hardware but is not exercised/ tied off as all partitions' power gating actions are triggered on L2 warm reset assertion. Similarly, the SC7_REMOVE_SLEEP state is also not used. Pmc2car_ram_awake signal still goes LOW during entry and de-asserted on exit to keep the reset sequencing.

- PMC Programming Guidelines
- The hardware sequencer asserts the global SEL_DPD and after a programmable delay (in 32K
cycles), the global E_DPD is asserted. These global signals do not override the DDR pad controls. On the SC7 exit, once VDD_CORE is powered ON, the global E_DPD is de-asserted, and on expiry of the DPD timer, global SEL_DPD is de-asserted. The hardware sequencer proceeds through
- SC7_EARLY/MAIN_CLAMP, SC7_PWR_OFF states to enter SC7_ALLOW_AOPG state. A delay step
(RESIDENCY_THRESHOLD_CHECK) is implemented before entering SC7_ALLOW_AOPG state, so that potential “thrashing” of power gate/un-gate actions can be avoided for AO domain. Once in
- SC7_ALLOW_AOPG state, AOPM gets the indication (in the form of ao_pmc2pm_soc_wake_req = 0)
that AO power gating can be triggered.
- Once SC7 sequencer is in ALLOW_AOPG state, AOPM can request a power down of AO logic. When
such a request is seen by the PMC, the PG sequencer triggers AOPG actions. The AOPG sequencer (inside the PG controller module) will indicate the completion of power-gating actions to AOPM by asserting an ack. PMC is required to always respond to a power gating request with an ack.
- SC7_ALLOW_AOPG or SC7_OSC_OFF can be the destination state for SC7 transitions, depending
on whether shallow/deep dormant states are entered in AOPM. The SC7 exit is triggered when a valid wake (Tier 1 or Tier 2) is detected by the Wake engine. AOPM requests the PMC sequencer to reenable OSC and power ungates the AO domain as required (if the wake events meant for SPE cluster are triggered).
- The PMC sequencer prioritizes wake events over a request to power off OSC, if they happen
at the same time. SC7 reset (all warm resets i.e., l*warm reset) seen by VDD_CORE domain is asserted by the SC7 sequencer prior to entering the PWR_ON state. The SC7 reset deassertion occurs in the REMOVE_RESET state, which is entered only after making sure that the ack is seen by the PMC and after all other actions (i.e., clamp deassertion, DPD, and SD removal) are completed. This check guarantees that the PMC never takes the SoC logic out of reset before ensuring that all power- saving steps taken by AOPM and PMC while in SC7 they are completely reversed and the transition back to active state is permitted. Two to three 32 kHz clock cycles before the SC7 reset assertion, the clock stop indication is asserted then de-asserted one 32 kHz post-SC7 reset assertion. This signal is used as an indication by AOCAR module to gate the clocks active during SC7 reset assertion during debug mode, to avoid glitches on the clocks, which are used by debug related modules in SoC.
- There is a PMC_IMPL_SC7_DEBUG_CTRL_0_GLITCH_FREE_CLKEN bit provided to enable/disable
assertion of this clock stop indication control. In functional SC7 mode, this bit should be set to 0, which implies legacy mode. In SC7 debug mode, this bit should be set to 1, when warm_rst_debug override is not set.

- PMC Programming Guidelines
- The SC7 sequencer does not support ASSERT_SLEEP and REMOVE_SLEEP states. On warm
reset assertion, all RAM and logic SLEEP control corresponding to SoC Partitions (except XUSB partitions) are forced to asserted state.
##### 4.2.3.1.1 Programming the OSC Pad Controls
The OSC pad is under the CAR control as L1 reset default. Software programs the PMC override register (PMC_IMPL_OSC_EDPD_OVER_0_OSC_CTRL_SELECT->1) to transfer control to the PMC. Once the control is transferred to the PMC, it continues to be under PMC control until the next L0/ L1 reset. The CLK_OK pin masks the mode detection mechanism in the pad after power-on-reset in order to mitigate risk when the AVDD rail of the pad is later cycled during an SC7 transition to save power. CLK_OK should not be changed from its default of 0 to avoid the pad mode from changing to bypass to crystal during a reset sequence. If the AVDD_OSC rail is not powered off and back on during a chip reset sequence, CLK_OK must be held at 0 for the bypass mode to work. Refer to the PMC_IMPL_OSC_EDPD_OVER register.
##### 4.2.3.1.2 Powering Down the OSC
The SC7 hardware sequencer proceeds to OSC_OFF state only if it is right to turn off OSC in SC7. The decision to turn off OSC is dependent on the following factors. No software overrides bits that disables OSC power down in SC7 are set. (PMC_IMPL_OSC_EDPD_OVER.XO_LP0_MODE). The AOPM signals to the PMC that OSC can be shut off. The high-level state of OSC_OFF consists of the following hardware sequence on powering down the OSC: 1.
- Hand shake with the TSC to ensure that the switch over to the 32 kHz clock is complete
within the TSC. The PMC sends a clock switch request to MTSC, which is acknowledged by
- TSC by asserting an ack to PMC. Note: The TSC is required to acknowledge the switch
request irrespective of the TSC configuration settings. Assert an OSC stable indication to cleanly gate off the undivided OSC clock within AO CAR. This step is required to ensure that there are no clock glitches propagated to active logic when the OSC pad is disabled. The CAR clock gating logic uses this OSC stable indication for clock gating. De-assert EN_XO to OSC pad after at least one 32 kHz clock cycle. 2. 3. When the SC7 state is exited, following sequence is required to be done in OSC_ON state. 1. Assert EN_XO to reenable the OSC pad.

#### 4.2.3.2 Wake Engine Programming
##### 4.2.3.2.1 Wake Configuration
The wake engine supports the following wake-specific configuration: 1.
- Wake-signaling parameters: These include the wake polarity, filter/de-bounce settings as
applicable. These signaling parameters could be platform-dependent and need to comprehend specific wake signal properties (for example, CAN wakes)
- Wake coalescing configuration (coalescing enable, coalescing group selection)
- Wake masks and wake status indication
- Wake routing configuration
2. 3. 4. The wake-specific configuration handled in software, for the wake engine to detect wakeup events and trigger SC7 exit is as follows: 1. 2. 3. 4. 5. 6. Enable signaling parameters for all wakes. Enable coalescing on all Tier1 and Tier2 wakes. Program group 0 timer with a value that meets PMIC sequencing requirements. Map all Tier1 and Tier2 wakes to group 0 window. Route wake events to any of three tiers. Unmask wake events.
##### 4.2.3.2.2 Delay Buffer Selection
A programmable number of delay chains were used to implement the delay filter logic. In the worst case, the requirement must reach at least ~2 ns delay to support glitch filtering. Before wake is enabled, the WAKE_AOWAKE_DELAY_CELL_CTRL_0_SIG_DELAY_BUF_SEL register must be configured with a tap select value based on per tap delays. This register must be programmed to 14 (hex value: E) to meet 2 ns delay.
##### 4.2.3.2.3 Wake Coalescing
The wake coalescing logic has hooks for software to enable/disable coalescing, to map wakes to different groups, and to program group timers to hold wakes based on wake latency tolerance.
- Enabling coalescing and group selection is handled through WAKE_AOWAKE_CTRL_x, which is
assigned per wake event. There are global timers for each group window. These timers can be programmed through WAKE_AOWAKE_COAL_GRP0/1_0.

##### 4.2.3.2.4 Global Disable Control for Wake Coalescing
The coalescing output on each wake line is triggered when the corresponding coalescing window expires and wake status bit is set. Coalescing timers are implemented so that they are free running only in SC7 i.e., across PWR_OFF and PWR_ON state of the SC7 sequencer. Any wake that is triggered after PWR_ON state or in SC0 is not seen on the interrupt lines until coalescing is disabled. The WAKE_AOWAKE_CNTRL_0_GLOBAL_COAL_EN bit can be used to disable coalescing on all wakes on SC7 exit. This bit should be LOW in order to do this. Individual COAL_EN takes effect when set to 1. However, wake status registers indicate the right wake-up status of all wake events occurred until then. In SC0, i.e., while the system is in active state, if a wake interrupt needs to be detected: WAKE_DET_EN should be enabled. Coalescing must be disabled on all wakes.
