# 10.7.1 Overview

@{Key=10.7.1; Title=Overview; Body=IEEE 802.1AE Media Access Control Security (MACsec) is a standard, defined to provide secure communications between stations that are attached to the same LAN. MACsec can identify and prevent most security threats, including denial of service, intrusion, man-in-the-middle, masquerading, passive wiretapping, and playback attacks. IEEE 802.1AE defines the data plane of the MACsec. This standard defines the frame format and mechanism of encryption, authentication, and replay protection. MACsec Key Agreement (MKA), a key management protocol that provides a key generation mechanism, is defined in IEEE 802.1X-2010 [3] . MACsec uses Secure Association Keys (SAKs) derived from MKA negotiation to perform integrity check on the data and optionally perform encryption. In the platforms, some of the use cases of MACsec (Integrity check only) are highlighted below.
- Network control traffic
Detection of False Address Resolution Protocol (ARP) replies.
- Inter-ECU traffic
Authentication of Inter-ECU packets to prevent middle-man attacks.
- Sensor control and data traffic
Authentication of control and data traffic from sensors. MACsec provides security at an OSI layer2 network. Some of the high-level features provided by MACsec are: Provides data confidentiality by providing strong encryption at OSI layer2. Provides cryptographic integrity checking to ensure that data cannot be modified in transit.
- Can be selectively enabled using a centralized policy to help ensure that it is enforced
where required while permitting non-MACsec-capable components to access the network. Encrypts packets between two MACsec enabled Ethernet ports.
#### 10.7.1.1 Limitations
The MACsec function is not available in the 10/100 Mbps speed modes on the 1G Ethernet RGMII port; it is only supported at 1 Gbps.

- MAC Security (MACsec)
#### 10.7.1.2 Controlled and Uncontrolled Ports
The MAC security entity (SecY) as specified in 802.1AE, defines two logical port types known as ‘controlled port’ and uncontrolled port on the same physical port. An uncontrolled port is used to exchange messages between authenticator and authentication server to set up secure connection. The controlled port is enabled when secured connections are established between the end points. The controlled port is used to exchange secured messages between authenticated end points.
#### 10.7.1.3 MACsec Key Agreement Protocol
The MACsec Key Agreement Protocol (MKA) is used for discovering MACsec peers and negotiating keys between MACsec peers. MKA relies on IEEE 802.1X Extensible Authentication Protocol (EAP) [3] framework to establish communication.
- MACsec peers belong to a unique connectivity association. Members of the same connectivity
association identify themselves with a shared Connectivity Association Key (CAK). This key can be statically preshared key or dynamically derived. CAK is the main key that is used to generate all other keys that are used for MACsec. The MKA protocol uses CAK to derive the Secure Association Keys (SAKs). These SAKs are used to establish the secure channel across MACsec peers. All peers in same connection association (CA) use the same cipher suite, either GCM-AES-128 or GCM-AES-256. The detail architecture of key management is outside the scope of this documentation. Refer to the IEEE STD 802.1X [3] for more information.

MACsec Frame fields are defined as: MAC-DA/MAC-SA—Destination and source MAC addresses, each occupying six octets. SecTAG—Security tag, which contains 8-16 octets. For detailed description, see below.
- Payload— Original Ethernet Frame fields that will be authenticated. These fields contain
original EtherType, VLAN, and frame payload (the bytes after the SecTAG and before the ICV). ICV—Integrity check value, which contains 16 octets.
- FCS - The frame check sequence is a four-octet cyclic redundancy check (CRC) that detects
corrupted data within the entire frame as received on the receiver side. SecTAG fields are broken into subfields as:
- MACsec EtherType–Contains MACsec EtherType value, which should be 0x88e5. If EtherType
is not 0x88e5, then packet is not identified as MACsec packet.
- TCI—TAG control information, which contains 6 bits to provide the following status
information. Table 10-1 in 802.1AE IEEE standard lists out all possible valid combinations of bits in SECTAG field. V–Version number. Default value is 0.

- MAC Security (MACsec)
ES–Indicates the end station. If set to 1, then source MAC address is used as first 6 bytes of SCI. SC–Secure Channel. If SCI is present in SecTAG, then this bit is set to 1.
- SCB–Not relevant
E–Data payload is encrypted. This bit is 0. C–Indicates Changed Text. This bit is 0.
- AN–Association number. It contains 2 bits and indicates the secure association (SA) number
of the secure channel (SC) on which the frame is sent. One SC can have four SAs (secure associations). SL–Short length. The field value is the number of the octets in the data field if the number of octets in the Secure data field is less than 48. Otherwise, the value of this field is zero.
- PN–Packet number, which contains four octets. The field value increments by 1 every time a
frame is sent. This field is used to guard against the replay attack. SCI–Secure channel identifier, which contains eight octets. This is an optional field, and it exists only when SC bit in TCI field is set to 1.
#### 10.7.1.4 MACsec Flow
- When the MACsec session is established between two MAC ports, all controlled port traffic
between intended source and destination MAC port, is MACsec-secured using data integrity checks and, if configured, encryption. The MACSEC block does not support Encryption though. MACsec inserts an 8-16 bytes secTAG header and appends 16 bytes of ICV to all Ethernet frames being transmitted by the MACsec-secured Ethernet link. The receiveing MACsec interface check the SecTAG header and ICV to ensure that the data was not compromised while traversing the link. If the data integrity check detects anything irregular about the packet, the packet is dropped, and an error is logged. If number of ICV errors exceed a threshold, an interrupt gets generated.

#### 10.7.1.9 Software Access View of MACsec IP
Three distinct software entities manage the MACsec: 1. The SecY implementation consists of two components:
- SecY software component: This is the part of the MACsec functional driver that programs
various MACsec protocol parameters and collects statistics and status. Since the MACsec block is common to various virtual endpoints sharing the MAC, the MACsec SecY is typically a common driver managing the service.
- SecY Hardware Component: Majority of SecY functionality is implemented in
- MACsec hardware with SecY software component programming the SecY hardware as to
which traffic flows over the controlled and uncontrolled path, its parameters, and related statistics. a. b.
- KaY is the Key Management Entity entirely implemented in software and manages various
- Cryptographic Keys (such as Generation and refreshing), Initialization Vectors, and
algorithmic parameters related to the ICV operation. 2.

- MAC Security (MACsec)

#### 10.7.1.10 AMAP View of MACsec IP
From the programming perspective of the address map, the MACsec IP space is mapped into 3x64
- KB pages. One 64 KB space is mapped to TZ software, which is used for programming MACsec
- Keys. The SecY software handles the other two 64 KB pages. CBB BLFs provide the isolation
control.
